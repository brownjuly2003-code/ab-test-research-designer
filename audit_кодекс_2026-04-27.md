## Executive Summary

AB_TEST выглядит как зрелый portfolio/demo full-stack проект: FastAPI + React 19 + Vite, локальное SQLite-хранилище, опциональный Postgres, OpenAPI -> TS contract generation, rich docs, a11y tests, smoke/e2e, Lighthouse, Docker/GHCR workflow, API-key/webhook/Slack hardening.

Главный риск не в "грязном коде", а в доверии к результатам и воспроизводимости release gates:

- P0/P1: continuous post-test выдает p-value/CI через normal approximation, хотя код делает вид, что использует Welch Student t.
- P1: frontend unit/a11y suite локально не проходит из-за таймаутов jsdom/axe/Recharts.
- P1: backend pytest по умолчанию на Windows ломается из-за temp permissions; с `--basetemp` проходит.

## Findings

### P0/P1 — Continuous post-test math uses normal approximation instead of Student t

Evidence:

- `app/backend/app/services/results_service.py:106` computes Welch degrees of freedom.
- `app/backend/app/services/results_service.py:107` calls `t_cdf(...)`.
- `app/backend/app/services/results_service.py:138` defines `t_cdf(value, df)`.
- `app/backend/app/services/results_service.py:140-141` returns `standard_normal_cdf(value)` even when `df` is finite.
- `app/backend/app/services/results_service.py:108` uses `normal_ppf` for continuous CI.
- `app/backend/app/services/results_service.py:124` hardcodes continuous `power_achieved=0.0`.

Impact: for small/medium samples, continuous observed results can be called significant too aggressively and show too-narrow confidence intervals. For an A/B testing product, this is decision correctness, not cosmetic math.

Fix direction: add a red test with low Welch df and tabulated expected p-value/CI, then either add a well-tested Student t implementation or a dependency-backed implementation. Also decide whether `power_achieved` should be computed or omitted/renamed for continuous results.

### P1 — Frontend unit/a11y suite is locally unstable

Evidence:

- `npm run test:unit` failed on local timeouts.
- `app/frontend/src/test/a11y-results.test.tsx:87` has explicit `30000`.
- `app/frontend/src/test/a11y-comparison-distribution-view.test.tsx:148` has explicit `30000`.
- `app/frontend/src/test/a11y-wizard.test.tsx:118-223` has multiple explicit `15000`.
- `app/frontend/src/test/a11y-sidebar.test.tsx:184-344` has multiple explicit `15000`.
- `app/frontend/src/components/PosteriorPlot.test.tsx:92` has explicit `15000`.
- `app/frontend/src/test/setup.ts` only stubs locale fetch; canvas/ResizeObserver stubs are repeated per file.

Impact: CI confidence erodes; small UI changes will be hard to land safely. Axe coverage is valuable, but currently too expensive/flaky for the default unit suite on this machine.

Fix direction: centralize jsdom browser stubs in `src/test/setup.ts`, split heavy axe coverage from core unit tests or make it deterministic, reduce repeated full-shell renders, and keep one real-Recharts integration test separate from flat-stub a11y tests.

### P1 — Backend pytest default command fails on Windows temp permissions

Evidence:

- Default backend command failed with `PermissionError` under user temp.
- Same suite passed with explicit `--basetemp .tmp_codex_ab_audit_pytest2_20260427`.

Impact: local Windows contributors cannot trust the documented/default pytest command, despite the backend suite itself being green with a controlled temp root.

Fix direction: configure pytest/verify scripts to use a repo-local temp root that is cleaned or ignored, then re-run default command without manual flags.

### P1 — Workspace import lacks explicit write transaction

Evidence:

- `app/backend/app/repository.py:2462` defines `import_workspace`.
- `app/backend/app/repository.py:2487` opens `with self._connect() as connection`.
- Existing write probe uses `BEGIN IMMEDIATE` at `app/backend/app/repository.py:2223`.

Impact: Python sqlite starts deferred transactions by default. Large workspace import can race with other writes and may not reserve the write lock early enough. For backup/restore, partial/racy behavior is worse than a clean busy failure.

Fix direction: add focused atomicity/concurrency tests, then execute `BEGIN IMMEDIATE` at import start for SQLite. Validate Postgres translation.

### P1 — Postgres backend is too much SQLite inheritance plus string translation

Evidence:

- `app/backend/app/repository.py:2711` has `class PostgresBackend(SQLiteBackend)`.
- `app/backend/app/repository.py:2775-2776` does SQL translation via `replace("BEGIN IMMEDIATE", "BEGIN")`; the implementation also replaces placeholders.
- `.github/workflows/test.yml:100-101` runs only `test_postgres_backend.py` smoke coverage for Postgres.

Impact: any new SQLite-specific query can silently escape review and fail only on Postgres paths not covered by smoke. This is a strategic risk if Postgres is marketed as production-capable.

Fix direction: before splitting `repository.py`, create a shared repository contract test suite that runs on SQLite and Postgres for workspace import/export, audit log, webhooks, Slack installation, API keys, and query filters. Then extract dialect-specific queries intentionally.

### P1 — Frontend i18n parity drift after Slack App UI

- `ar/de/es/fr/zh` miss 12 `sidebarPanel.slackApp.*` keys relative to English.
- Backend i18n parity is clean: all locales `235`.

### P1/P2 — `.env.example` omits `AB_ADMIN_TOKEN`

Evidence:

- `.env.example:26-27` contains `AB_API_TOKEN` and `AB_READONLY_API_TOKEN`.
- README documents managed API keys via `AB_ADMIN_TOKEN` at `README.md:218`.
- Release notes also require `AB_ADMIN_TOKEN` for webhook CRUD at `docs/RELEASE_NOTES_v1.1.0.md:29`.

Impact: self-hosted secure setup is incomplete by copy-paste. Admin-only keys/webhooks can look broken unless the operator reads deeper docs.

Fix direction: add `AB_ADMIN_TOKEN=` to `.env.example` and align quickstart/deploy docs.

### P2 — `.env.example` is hardcoded to this Windows checkout

Evidence:

- `.env.example:4` has `AB_DB_PATH=D:\AB_TEST\app\backend\data\projects.sqlite3`.
- `.env.example:11` has `AB_FRONTEND_DIST_PATH=D:\AB_TEST\app\frontend\dist`.

Impact: poor onboarding on Linux/macOS/Docker and any Windows clone not located at `D:\AB_TEST`.

Fix direction: use relative paths or comments with OS-specific examples.

### P2 — Sync persistence is used in async request hot paths

Evidence:

- `app/backend/app/http_runtime.py:195` authenticates DB API keys in middleware.
- `app/backend/app/http_runtime.py:209` logs API-key use before request rate limiting at `app/backend/app/http_runtime.py:231`.
- `app/backend/app/repository.py:81` uses synchronous `sqlite3.connect`.

Impact: for local-first/single-user this is tolerable; for hosted concurrent usage it blocks the event loop and records audit entries even for requests that may then be rate-limited.

Fix direction: move rate limit before audit logging where possible, batch/sample API-key-used audit logs, or put repository calls behind `asyncio.to_thread`/async drivers for hot paths.

### P2 — In-memory rate limiter can grow by unique key

Evidence:

- `app/backend/app/http_utils.py:54` stores `_events: dict[str, deque[float]]`.
- `app/backend/app/http_utils.py:68` uses `setdefault`.
- Empty/expired buckets are not removed.

Impact: long-lived public deployments can accumulate keys from rotating IPs or spoofed forwarded headers. Severity is medium for demo runtimes that restart often, higher for Fly/long-running public deployments.

Fix direction: delete buckets after expiry when empty and add a max-key guard or periodic prune.

### P2 — Slack OAuth tokens are stored plaintext

Evidence:

- `app/backend/app/repository.py:312-313` has `bot_token TEXT NOT NULL`, `user_token TEXT`.
- `app/backend/app/repository.py:1057-1072` upserts Slack tokens.

Impact: local SQLite compromise exposes Slack bot/user tokens. This may be acceptable for local demo mode, but hosted mode needs a clear posture.

Fix direction: either document local-only plaintext storage and rotation guidance, or encrypt tokens at rest with a required deployment secret.

### P2 — Snapshot loop has no outer guard

Evidence:

- `app/backend/app/main.py:137-140` runs a background snapshot loop and awaits `snapshot_service.push_snapshot()`.
- Final shutdown push is guarded at `app/backend/app/main.py:152`.

Impact: current `SnapshotService.push_snapshot()` catches most expected failures, so this is not urgent. But a future uncaught bug will kill the background task without restarting it.

Fix direction: wrap each loop iteration in `try/except Exception`, log, and continue.

### P2/P3 — OpenAPI metadata and dependency warning drift

Evidence:

- README/license badge says MIT at `README.md:9` and `README.md:16`.
- FastAPI metadata says `license_info={"name": "UNLICENSED"}` at `app/backend/app/main.py:171`.
- Contract/docs generation emits `email-validator not installed, email fields will be treated as str`; contact email is set at `app/backend/app/main.py:170`.

Impact: generated API docs send mixed legal metadata and noisy warnings.

Fix direction: set OpenAPI license to MIT and either add `email-validator` or remove the contact email metadata if it is only placeholder data.

### P3 — Docs-site page is built but not in nav

Evidence:

- `python -m mkdocs build --strict` passed but reported `docs-site/features/database.md` exists outside `nav`.
- `docs-site/features/database.md` documents SQLite/Postgres selection, which is relevant to current architecture risk.

Impact: useful operational docs are easy to miss.

Fix direction: add Database to the `Features` or `Reference` nav in `mkdocs.yml`.

### P3 — Large duplicate screenshot assets

Evidence:

- `docs/demo/comparison-dashboard.png` and `docs-site/assets/screenshots/comparison-dashboard.png` are both about `1.59 MB`.
- `docs/demo/results-dashboard.png` and docs-site copy are both about `1.10 MB`.
- `docs-site/assets/screenshots/comparison-distribution-view.png` is about `2.05 MB`.

Impact: clone size and screenshot churn grow unnecessarily.

Fix direction: choose one screenshot source of truth or add an asset sync/compression step.

### P3 — Local test artifacts are large

Evidence:

- `app/backend/tests/.tmp`: `10 803` files, `999 175 030` bytes during audit.
- Additional audit pytest temp dirs exist: `.tmp_codex_ab_audit_pytest_20260427/`, `.tmp_codex_ab_audit_pytest2_20260427/`.

Impact: local disk/worktree noise; can slow recursive tooling.

Fix direction: standardize test temp location, ignore it, and provide cleanup script/target. Do not delete current artifacts blindly if another session depends on them.

## Corrected Non-Findings

- Recharts is not fully eager-loaded in production results views; major chart components are already lazy-loaded. Remaining issue is chunk/test cost, not total lack of lazy loading.
- `calculate_cuped_variance_reduction()` ignores `pre_experiment_std`, but `variance_reduction = correlation**2` is valid if the input is already Corr(X,Y). Treat as API clarity/doc issue unless the intended input is covariance.
- No evidence found that API/LLM tokens are logged raw through normal structured logging; `logging_utils.py` masks `Authorization`, `X-API-Key`, and `X-AB-LLM-Token`.
- OpenAPI generated TS contract and `docs/API.md` are up to date.
