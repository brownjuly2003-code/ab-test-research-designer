# P1.2 — Live-stats denominator consistency (audit §5.4, finding D)

## Root cause (systematic-debugging, confirmed against the seeded Checkout demo)

Two independent bugs produce the audit's "4000 из 3992" / "guardrail 2001 vs primary 2000":

1. **Rate-spike bot detection is metric-scoped, not experiment-scoped.**
   `get_experiment_analysis_aggregates(experiment_id, metric_name)` excludes a user as a
   rate-spike bot only when their event count *on that one metric* exceeds
   `BOT_CONVERSION_EVENT_THRESHOLD`. The demo's bot (`ckt-bot-1`) fires 130 events on the
   *primary* metric only, so it is removed from the primary rollup (2001 → 2000) but not from
   any guardrail rollup — the guardrail calls the same function with its own metric name, sees
   zero events from the bot on *its* metric, and keeps it (2001). Two rollups over the same
   exposed population report different N depending on which metric happened to catch the spike.

2. **`get_stratified_aggregates` has no identity resolution and no exclusion filter at all** —
   it joins raw `exposures`/`user_strata` directly on `user_id`, unlike every other rollup which
   folds anonymous→canonical (`identity_map`) and drops manual/rate-spike exclusions
   (`excluded_users`). In the demo this makes `stratified_users_total` (4000, the raw
   canonical-user count with a recorded stratum) exceed `exposed_users_total` (3992, the primary
   rollup's fully resolved+filtered count) — a "4000 of 3992" that reads as nonsense on the page.

Traced by reproducing the demo's exact numbers by hand from `demo_execution.build_checkout_execution`
(2000/arm + 1 bot + 60 anon-linked duplicates + 8 manual exclusions) against the SQL in
`repository.py` — the arithmetic matches the audit's reported figures exactly, confirming both
mechanisms independently before touching any code.

## Fix (root cause, not a label)

- Rate-spike is a property of the *user* (an automation/instrumentation artifact), not of one
  metric's event stream — compute it once per experiment over **all** the experiment's
  conversions, and reuse that same set in every per-metric rollup. This makes guardrail and
  primary `exposed_users` identical for the same arm, on any metric.
- `get_stratified_aggregates` gets the same identity-fold + exclusion filter as
  `get_experiment_analysis_aggregates` (mirrored CTE shape), so `stratified_users_total` is a
  subset of `exposed_users_total` by construction — the only remaining gap is genuinely "no
  stratum recorded," which is what the UI copy already says.
- `get_exclusion_summary`'s rate-spike count switches to the same global definition so the
  indicator matches what the rollups actually did.
- Out of scope (deliberately): holdout / CUPED / ratio aggregates keep their own, already-labeled
  populations (held-back group, covariate-complete subset) — the audit only flagged the
  stratified and guardrail denominators, and those have their own honest notes already.

## Verify

- New tests: guardrail vs primary `exposed_users` equal when a bot spikes only the primary
  metric; stratified rollup drops an identity-linked duplicate and a manually-excluded user the
  same way the primary rollup does.
- Full backend gate (ruff, mypy --strict, pytest, contract --check).
- Live demo Checkout: `/live-stats` guardrail block shows the same N as the primary comparison
  block; stratified coverage line no longer shows a numerator above its denominator.
